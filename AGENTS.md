# AGENTS.md

## 프로젝트 목적

이 프로젝트는 브라우저만으로 실행 가능한 `3D 디지털 수족관`을 만드는 것을 목적으로 한다.

핵심 목표는 아래와 같다.

- 관상용 앱으로서 `심미성`, `몰입감`, `차분한 감상 경험`을 최우선으로 한다.
- 사용자 피드백상 `물고기/유목/수초/바위의 현실감`이 부족하면 다른 연출보다 먼저 해결한다.
- 별도 백엔드 없이 `정적 파일만으로 실행 가능한 구조`를 유지한다.
- 수초, 유목, 바위, 물고기, 기포, 조명, 카메라 움직임이 하나의 조화로운 풍경으로 느껴지게 한다.
- 기능 추가보다 먼저 `구도`, `조명`, `색감`, `움직임`, `성능`을 다듬는다.

현재 앱의 핵심 경험은 다음과 같다.

- 3D 아쿠아스케이프 감상
- 물고기 유영과 수초/기포 애니메이션
- 조명 조절
- 오브젝트 클릭 시 명언 표시

## 에이전트의 기본 동작

이 프로젝트에서 작업하는 에이전트는 아래 순서로 행동한다.

1. 먼저 현재 워크스페이스 구조와 실제 파일 상태를 확인한다.
2. 작업을 시작하기 전에 반드시 `AGENTS.md`, `PLAN.md`, `PROGRESS.md`를 읽는다.
3. 조사나 검색이 필요한 작업이면 `references/` 폴더의 기존 문서를 먼저 확인한다.
4. `PLAN.md`에서 현재 우선순위와 미완료 작업을 확인하고, 사용자 요청과 연결되는 작업이 있으면 이어서 수행한다.
5. 요청이 구현을 요구하면 설명만 하지 말고 가능한 범위에서 직접 수정한다.
6. 사용자의 직접 피드백이 있으면 기존 우선순위보다 먼저 반영한다.
7. 변경 전후로 `관상용 심미성`에 어떤 영향이 있는지 판단한다.
8. 특히 생물/하드스케이프 구현에서는 아래 순서를 우선한다.
   - 물고기 실루엣과 유영의 생물감
   - 유목의 밀도, 분지 구조, 활착 표현
   - 수초/바위의 질감과 반복 패턴 완화
9. 시각적 변경이 있으면 가능하면 브라우저에서 직접 확인하거나 캡처를 남긴다.
10. 작업이 끝나면 `PLAN.md`의 상태를 갱신하고, `PROGRESS.md`에 완료 내용과 다음 인수인계 메모를 남긴다.
11. 구조를 바꾸면 `AGENTS.md`도 함께 갱신한다.

## 협업 문서 운영 규칙

멀티 에이전트 협업 시 아래 세 문서를 기준으로 작업을 이어간다.

- `AGENTS.md`
  - 프로젝트의 고정 규칙과 운영 원칙
- `PLAN.md`
  - 앞으로 해야 할 작업, 우선순위, 상태
- `PROGRESS.md`
  - 완료 이력, 작업 로그, 다음 작업자가 이어받아야 할 정보
- `references/`
  - 외부 조사 결과, 링크, 요약, 프로젝트 적용 메모
- `skills/`
  - 프로젝트 전용 스킬 정의

### 시작 전 필수 절차

- 새 작업을 시작할 때는 반드시 아래 순서를 따른다.
  1. `AGENTS.md` 읽기
  2. `PLAN.md` 읽기
  3. `PROGRESS.md` 읽기
  4. 조사성 작업이면 `references/` 확인
  5. 현재 요청이 어느 작업과 연결되는지 판단

### references/ 운영 규칙

- 웹 검색이나 레퍼런스 조사를 했다면 결과를 `references/` 폴더에 남긴다.
- 원문 전문을 저장하지 말고 아래 형식으로 요약한다.
  - 조사 날짜
  - 주제
  - 핵심 인사이트
  - 프로젝트 적용 메모
  - 출처 URL
- 이미 같은 주제 문서가 있으면 새 파일을 만들기보다 기존 문서를 갱신한다.
- 조사 결과를 바탕으로 계획이 바뀌면 `PLAN.md`도 함께 갱신한다.

### skills/ 운영 규칙

- 프로젝트 전용 반복 작업 흐름이 생기면 `skills/`에 스킬로 정의한다.
- 스킬은 가능한 한 재사용 가능한 절차, 라우팅 기준, 문서 갱신 규칙을 담아야 한다.
- 프로젝트 스킬을 수정했으면 관련 규칙 변경이 `AGENTS.md`와 충돌하지 않는지 확인한다.
- 새 에이전트가 프로젝트를 이어받을 때는 `skills/aquagarden-project/`도 참고 대상으로 삼는다.

### PLAN.md 작성 규칙

- 각 작업은 고유 작업 ID를 가진다. 예: `AQ-101`
- 상태는 아래 네 가지 중 하나만 사용한다.
  - `todo`
  - `in_progress`
  - `blocked`
  - `done`
- 우선순위는 `P1`, `P2`, `P3`로 표기한다.
- 새 작업을 추가할 때는 `작업 내용`, `완료 기준`, `비고`를 함께 적는다.
- 이미 존재하는 작업과 겹치면 새 항목을 만들지 말고 기존 항목을 업데이트한다.

### PROGRESS.md 작성 규칙

- 완료 기록은 날짜 기준으로 누적한다.
- 각 기록에는 아래 내용을 포함한다.
  - 작업 ID
  - 무엇을 완료했는지
  - 어떤 파일이 바뀌었는지
  - 확인/검증 결과
  - 다음 작업자가 바로 이어서 할 수 있는 메모
- 기존 기록을 지우기보다 `추가 기록` 방식으로 남긴다.

### 인수인계 규칙

- 작업을 끝내지 못한 상태에서 멈출 경우, `PROGRESS.md`에 `중단 지점`과 `다음 액션`을 반드시 남긴다.
- `PLAN.md`의 상태와 `PROGRESS.md`의 메모가 서로 모순되지 않도록 유지한다.
- 사용자 요청으로 계획이 바뀌면 먼저 `PLAN.md`를 갱신한 뒤 구현을 진행한다.

## 에이전트가 따라야 하는 기본 룰

### 1. 제품 방향성

- 이 앱은 `게임`보다 `감상용 디지털 오브제`에 가깝다.
- 기능을 추가할 때는 항상 아래 우선순위를 따른다.

```text
심미성 > 몰입감 > 부드러운 움직임 > 성능 > 기능 수
```

- 수족관이 복잡해 보이는 것보다 `정돈되고 깊이 있어 보이는 것`이 더 중요하다.
- 과장된 연출보다 `차분하고 자연스러운 연출`을 선호한다.

### 2. 기술 방향

- 기본 전제는 `브라우저 단독 실행`이다.
- 특별한 이유가 없으면 백엔드, 데이터베이스, 서버 렌더링을 도입하지 않는다.
- 특별한 이유가 없으면 React, Vue, 대형 프레임워크로 전환하지 않는다.
- 특별한 이유가 없으면 빌드 도구 없이도 열리는 구조를 우선한다.
- Three.js 사용 시 공식 모듈과 표준 ES module 방식을 우선한다.

### 3. 코드 구조

- 현재는 단일 HTML 파일 구조를 유지한다.
- 코드가 커져서 관리가 어려워질 때만 모듈 분리를 제안한다.
- 파일 분리를 할 경우 `scene`, `entities`, `systems`, `ui`, `data` 기준으로 나눈다.
- 수치 상수는 흩어놓지 말고 가능한 한 한 곳에 모은다.
- 랜덤 기반 배치도 결과가 너무 산만하지 않도록 범위를 제어한다.

### 4. 시각 디자인 룰

- 조명은 전체를 균일하게 밝히기보다 `주광 + 보조광 + 그림자 대비`를 활용한다.
- 색상은 많이 쓰기보다 `절제된 팔레트`를 유지한다.
- 붉은색이나 고채도 포인트는 제한적으로 사용한다.
- 수초는 랜덤하게 흩뿌리기보다 `군집`, `레이어`, `시선 유도`를 고려한다.
- 물고기는 풍경을 압도하지 않게 유지한다.
- UI는 감상을 방해하지 않도록 작고 조용하게 유지한다.

### 5. 애니메이션 룰

- 물고기 움직임은 빠르고 시끄럽기보다 `유영감`, `관성`, `완급`이 느껴져야 한다.
- 수초는 같은 패턴으로 흔들리지 않게 위상과 강도를 다르게 준다.
- 기포는 기계적으로 동일하게 올라가지 않도록 크기, 속도, 흔들림을 분산한다.
- 카메라와 조명 애니메이션은 과하지 않게 유지한다.

### 6. 상호작용 룰

- 클릭 인터랙션은 감상 흐름을 깨지 않아야 한다.
- 명언 패널은 정보량보다 분위기 유지가 우선이다.
- 조명 컨트롤은 직관적이어야 하며 작은 조정만으로도 분위기 차이가 느껴져야 한다.

### 7. 성능 룰

- 데스크톱 브라우저에서 부드럽게 동작하는 것을 기본 기준으로 삼는다.
- 오브젝트 수가 늘어나면 `Instancing`, `LOD`, 텍스처 최적화부터 검토한다.
- 시각 효과를 추가할 때는 먼저 성능 비용을 확인한다.
- 시각적으로 큰 차이가 없으면 더 단순한 구현을 선택한다.

### 8. 검증 룰

- 변경 후 최소한 아래 항목을 확인한다.
  - 브라우저에서 정상 로드되는가
  - 콘솔 에러가 없는가
  - 카메라 조작이 되는가
  - 조명 UI가 반응하는가
  - 물고기, 수초, 기포 애니메이션이 유지되는가
  - 모바일 폭에서도 UI가 깨지지 않는가
- 시각적인 변경이 크면 스크린샷이나 미리보기 파일을 남긴다.
- 검증을 마쳤으면 결과를 `PROGRESS.md`에도 짧게 남긴다.

### 9. 변경 금지/주의 사항

- 사용자가 요청하지 않은 큰 기술 스택 전환은 하지 않는다.
- 사용자가 요청하지 않은 파일 삭제나 이름 변경은 하지 않는다.
- 스크린샷 참고 이미지는 임의로 지우지 않는다.
- 외부 에셋을 추가할 때는 출처와 라이선스를 확인한다.
- 검색 기반 제안을 할 때는 최신 정보가 필요한 경우 반드시 확인한다.

## 작업 판단 기준

사용자가 개선을 요청할 때 에이전트는 먼저 아래 질문을 스스로 점검한다.

1. 이 변경이 수족관을 더 아름답게 보이게 하는가
2. 이 변경이 감상 경험을 더 차분하게 만드는가
3. 이 변경이 구조를 불필요하게 복잡하게 만드는가
4. 같은 효과를 더 단순한 방식으로 낼 수 있는가

위 질문 중 1, 2는 만족하고 3, 4에서 문제가 적은 방향을 우선 채택한다.

## 완료 기준

작업은 아래 조건을 만족할 때 완료로 본다.

- 앱이 브라우저에서 실행된다.
- 핵심 감상 요소가 깨지지 않는다.
- 새 기능이 기존 분위기를 해치지 않는다.
- 구조 변경이 있었다면 이 문서가 최신 상태다.
